    备份与恢复¶

    本文介绍如何在分布式悦数图数据库集群中备份和恢复数据。

    分布式悦数图数据库集群的数据完整性依赖于 Meta 服务（管理集群拓扑、用户信息和元数据）与 Storage 服务（存储实际图数据）之间的协作。为了防止数据丢失，必须同时备份 Meta 服务和 Storage 服务。仅备份其中之一可能导致数据不可读或集群状态损坏。

    应用场景¶

    • 灾难恢复（数据丢失）：如果当前集群中的数据损坏或被意外删除，您可以将集群恢复到之前的健康状态。其涉及恢复 Meta 服务和 Storage 服务数据，该操作会覆盖当前主机上的现有数据。
    • 集群迁移或克隆：如果需要将数据移动到新主机或使用生产数据创建预发布环境，您可以将备份数据恢复到一个全新的服务组。可在不影响集群运行的情况下复制数据环境。
    • 日常数据保护：为了最大程度地减少崩溃事件中的数据丢失，您可以安排定期的全量和增量备份。这确保您始终拥有最近的恢复点。

    开始之前¶

    在执行备份和恢复操作之前，请查看以下概念、前提条件和使用限制：

    关键概念¶

    • 全量备份：将 Meta 服务或服务组的数据完整复制到备份目录，包括数据文件和预写日志 (WAL) 文件。
    • 增量备份：通过重放 WAL 来捕获自上次全量或增量备份以来的变更。增量备份与基准全量备份及后续的增量备份共同构成一个备份链。在恢复过程中，系统会从最新的增量备份回放至基准全量备份，以完全恢复数据。

    前提条件¶

    • 集群状态：悦数图数据库集群必须通过配置文件部署且运行正常。
    • 工具：必须部署 ngctl (5.2.0 版本或以上) 和 agent (5.1.0 版本或以上)。Agent 账户必须对备份路径具有读写权限。
    • 共享存储：必须将共享存储服务（例如 NFS）挂载到所有 Meta 和 Storage 主机的相同路径下（例如，Meta 主机为/data/backup/meta，Storage 主机为/data/backup/storage）。
    • 磁盘空间（仅限恢复）：对于恢复到原始集群的操作，主机必须有足够的可用空间（预留至少为当前数据大小再加 100 GB）。

    使用限制¶

    • 服务停机：恢复操作是离线进行的，期间服务不可用。请在非高峰时段执行此操作。
    • 向量索引：向量索引不会被备份。您必须使用 REPAIR INDEX 重新构建它们。更多信息，请参见 CREATE VECTOR INDEX 和 REPAIR INDEX。
    • 数据覆盖：恢复成功后，目标集群中的现有数据将被删除并替换为备份数据。
    • 拓扑匹配：恢复到新服务组时，新服务组必须具有与原始服务组相同的 Storage 服务拓扑和悦数图数据库版本。
    • 增量备份限制：
      • 不支持 Meta 服务的增量备份；仅支持服务组的增量备份。
      • 不支持基于早于 5.2.0 版本集群生成的全量备份进行增量备份。
      • 增量备份依赖于 WAL 的连续性。如果 WAL 被删除或中断，增量备份将失败。建议：
        • 修改raft_log_ttl_secs以配置 WAL 保留策略。更多信息，请参见 WAL 保留策略。
        • 定期执行全量备份，以减少恢复时间并降低链中断的风险。

    备份操作¶

    Note

    本文中的示例仅涵盖 Meta 服务和单个服务组的备份。对于包含多个服务组的集群，请对每个服务组依次执行备份；步骤相同。如需更详细的指导，请联系客户支持。

    1. 备份 Meta 服务（仅限全量备份）¶

    务必先备份 Meta 服务以确保元数据一致性。

    # 在 ngctl 所在的目录中运行
./ngctl meta backup -f config.yaml -p file:///data/backup/meta/full
    • config.yaml：用于部署原始集群的配置文件。
    • file:///data/backup/meta/full：原始集群中 Meta 服务主机的挂载点路径。该路径必须为空。

    更多信息，请参见备份 Meta 数据。

    2. 备份服务组（全量或增量备份）¶

    选项一：全量备份¶

    ./ngctl svcgrp backup ServiceGroup_A -f config.yaml -p file:///data/backup/storage/full
    • ServiceGroup_A：要备份的服务组名称。
    • file:///data/backup/storage/full：原始集群中 Storage 服务主机的挂载点路径。该路径必须为空。

    更多信息，请参见备份服务组数据。

    选项二：增量备份¶

    要执行增量备份，必须使用-i指向先前的基准备份（全量或增量备份）。

    ./ngctl svcgrp backup ServiceGroup_A -f config.yaml -p file:///data/backup/storage/incr -i file:///data/backup/storage/full

    增量备份与 WAL

    增量备份依赖于 Raft WAL 的连续性。如果 WAL 被删除或中断，增量备份将无法进行。例如，如果基准全量或增量备份后生成的 WAL 因 WAL 保留策略而被清除，导致 WAL 链断裂，增量备份将失败并报告类似以下的错误：

    Backup failed: [NF011]: Incremental backup failed: wal of part `1` is not consecutive
Error: [NF011]: Incremental backup failed: wal of part `1` is not consecutive

    为避免因 WAL 链不连续导致增量备份失败，建议检查并调整您的 WAL 保留策略配置。

    WAL 保留策略

    增量备份依赖于 Raft WAL 的连续性。如果 WAL 被删除或中断，将无法执行增量备份。

    WAL 保留由以下配置选项控制：

    选项 描述 默认值 数据类型 取值范围
    raft_log_ttl_secs WAL 生存时间（秒）。当此值大于 0 时，忽略 raft_log_maintain。
    建议：修改 raft_log_ttl_secs 的值，为 WAL 设置固定的生存时间。    		 0 int 0 ~ 2592000
    raft_log_maintain 保留的 WAL 条目数。仅当 raft_log_ttl_secs 为 0 时生效。清理周期由 raft_log_clear_timeout_ms 触发。 100000 int 0 ~ 4294967296
    raft_log_clear_timeout_ms 运行 WAL 清理任务的间隔（毫秒）。在每个间隔，系统会检查并删除满足以下任一条件的日志：
    • 按索引：当日志索引小于 applyIndex (当前应用索引) - raft_log_maintain (保留的 WAL 数) 时删除。
    • 按时间：当日志生存时间超过 raft_log_ttl_secs 时删除。
    		 600000 int -

    您可以在 Storage 服务配置文件 (nebula-storaged.conf) 中设置这些选项。有关更多信息，请参见基于文件的配置。

    3. 验证备份¶

    在原始集群的任一 Meta 或 Storage 服务主机上，检查对应备份目录。例如：

    • Meta 服务全量备份（位于 file:///data/backup/meta/full）的目录结构为：

      /data/backup/meta/full
├── backup.json   # 元数据索引文件
├── base          # 记录增量备份依赖关系的元数据指针。全量备份为空。
├── data          # 数据文件 
└── wal           # 预写日志 (Write-ahead logs)

    • 与全量备份不同，增量备份目录不包含 data 子目录。服务组增量备份（位于 /data/backup/storage/incr）的目录结构为：

      /data/backup/storage/incr
├── backup.json
├── base 
└── wal

    要查看详细的备份信息（如备份时间、名称、WAL 路径和范围等），请运行：

    cat backup.json | base64 --decode

    恢复操作¶

    Note

    如果需要将 Meta 服务数据恢复到新集群、恢复集群中的多个服务组或执行其他恢复场景，请联系客户支持。

    恢复到原始集群¶

    此场景在同一主机上恢复损坏的集群数据。

    1. 为了防止恢复失败导致原始集群数据丢失，请先对原始集群进行备份。更多信息，请参见备份操作。

    2. 基于全量备份恢复 Meta 服务数据（例如，位于 file:///data/backup/meta/full）。在 ngctl 所在的目录中，运行以下命令：

      Note

      当 Meta 服务数据损坏或 Meta 管理的用户信息在备份后发生变更时，须先运行此命令。

      ./ngctl meta restore -f config.yaml -p file:///data/backup/meta/full

      更多信息，请参见恢复 Meta 数据。

    3. 基于备份恢复服务组数据。例如，要从位于file:///data/backup/storage/incr的增量备份恢复服务组数据：

      ./ngctl svcgrp restore ServiceGroup_A -f config.yaml -p file:///data/backup/storage/incr

      更多信息，请参见恢复服务组数据。

    恢复到新服务组¶

    此场景将数据迁移到新环境。原始 Meta 服务必须正在运行，且新服务组必须具有与原始服务组相同的 Storage 服务拓扑和悦数图数据库版本。

    Note

    在此示例中，新服务组命名为 ServiceGroup_A_restore；Graph 服务运行在 192.168.15.252 和 192.168.15.253，Storage 服务运行在 192.168.15.252、192.168.15.253 和 192.168.15.254。步骤如下：

    1. 在每台新服务组主机上，安装并解压悦数图数据库包，并根据新服务组拓扑更新 Graph 和 Storage 服务配置文件中的 meta_server_addrs 和 host。

      在此示例中：

      • 在 192.168.15.252 上，更新 nebula-storaged.conf 和 nebula-graphd.conf。
      • 在 192.168.15.253 上，更新 nebula-storaged.conf 和 nebula-graphd.conf。
      • 在 192.168.15.254 上，更新 nebula-storaged.conf。

      有关安装和配置的信息，请参见步骤 1. 安装悦数图数据库 和步骤 2. 设置配置文件。

    2. 登录到原始集群的任意 Meta 服务（例如 192.168.15.249）：

      ./ngctl auth login --host 192.168.15.249 --port 9559 --user root --password NebulaGraph01

      更多信息，请参见登录 Meta。

    3. 基于备份恢复服务组数据（例如，位于file:///data/backup/storage/full的备份）：

      1. 创建新服务组(ServiceGroup_A_restore)：

        ./ngctl svcgrp create ServiceGroup_A_restore -r 3
        • 3：副本数（必须与原始服务组匹配）。

      2. （仅恢复多 Zone 服务组时需要）向服务组添加 Zone：

        基于部署多 Zone 服务组中的部署架构，以下示例将新集群的 Graph 和 Storage 服务分别放置在Zone1和Zone2两个 Zone 中：

        ./ngctl svcgrp add-zone ServiceGroup_A_restore --name Zone1 --priority 1 --replica-factor 2
./ngctl svcgrp add-zone ServiceGroup_A_restore --name Zone2 --priority 0 --replica-factor 1

        添加 Zone 后，可以运行 ./ngctl svcgrp show-zone ServiceGroup_A_restore 查看 Zone 信息。更多信息，请参见添加 Zone。

      3. 向服务组添加主机：

        • 对于单 Zone 服务组，运行：

          ./ngctl svcgrp add-host ServiceGroup_A_restore --host 192.168.15.252
./ngctl svcgrp add-host ServiceGroup_A_restore --host 192.168.15.253
./ngctl svcgrp add-host ServiceGroup_A_restore --host 192.168.15.254

        • 对于多 Zone 服务组，运行：

          ./ngctl svcgrp add-host ServiceGroup_A_restore --host 192.168.15.252 --zone Zone1
./ngctl svcgrp add-host ServiceGroup_A_restore --host 192.168.15.254 --zone Zone1
./ngctl svcgrp add-host ServiceGroup_A_restore --host 192.168.15.253 --zone Zone2

        执行上述命令后，可以运行 ./ngctl svcgrp show-host ServiceGroup_A_restore 查看主机信息。

      4. 向服务组添加服务：

        ./ngctl svcgrp add-service ServiceGroup_A_restore --type graphd --host 192.168.15.252 --port 9669
./ngctl svcgrp add-service ServiceGroup_A_restore --type graphd --host 192.168.15.253 --port 9669
./ngctl svcgrp add-service ServiceGroup_A_restore --type storaged --host 192.168.15.252 --port 9779
./ngctl svcgrp add-service ServiceGroup_A_restore --type storaged --host 192.168.15.253 --port 9779
./ngctl svcgrp add-service ServiceGroup_A_restore --type storaged --host 192.168.15.254 --port 9779

        执行上述命令后，可以运行 ./ngctl svcgrp show-service ServiceGroup_A_restore 查看服务信息。

      5. （仅恢复多 Zone 服务组时需要）设置 Zone 路由策略：

        ./ngctl svcgrp alter ServiceGroup_A_restore --route-policy zone_strict

        执行上述命令后，可以运行 ./ngctl svcgrp show ServiceGroup_A_restore 查看路由策略。更多信息，请参见更改 Zone 路由策略。

      6. 启动服务：

        ./ngctl svcgrp start-service ServiceGroup_A_restore -t storaged -f config_new.yaml
./ngctl svcgrp start-service ServiceGroup_A_restore -t graphd -f config_new.yaml

      7. 恢复服务组数据：

        Caution

        请指定正确的备份文件路径。如果路径不正确，恢复可能会失败。如果恢复失败，您需要从头开始重新启动恢复过程。

        ./ngctl svcgrp restore ServiceGroup_A_restore -f config_new.yaml -p file:///data/backup/storage/full

    验证恢复的数据¶

    1. 使用 ngql 连接到新服务组中的任意 Graph 服务（例如 192.168.15.252）：

      ./ngql -H 192.168.15.252

    2. 检查新服务组中的图数据：

      SHOW GRAPHS                                 # 列出所有图。
USE movie MATCH (v) RETURN v LIMIT 10       # 查询图 movie 中的数据。
SUBMIT JOB STATS movie                      # 对 movie 发起统计任务。
USE movie SHOW STATS                        # 显示 movie 的统计信息。
